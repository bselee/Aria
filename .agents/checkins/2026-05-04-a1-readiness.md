### Query 1 — Cluster candidates (the main signal)

Replace `<PASTE SQL HERE>` with:

```sql
SELECT
    at.source_table,
    at.type,
    COALESCE(at.inputs->>'vendor_name', at.inputs->>'agent_name', '(none)') AS vendor_or_source,
    th.event_type AS decision,
    count(*) AS n,
    ROUND(count(*) FILTER (WHERE th.status = 'success') * 1.0 / count(*), 3) AS agreement_rate,
    min(th.created_at) AS first_seen,
    max(th.created_at) AS last_seen
FROM task_history th
JOIN agent_task at ON at.id = th.task_id
WHERE th.created_at >= NOW() - INTERVAL '7 days'
  AND th.event_type IN ('approved', 'rejected', 'succeeded', 'failed')
  AND th.task_id IS NOT NULL
GROUP BY 1, 2, 3, 4
HAVING count(*) >= 5
ORDER BY n DESC;
```

Record: how many rows returned, and the `n` + `agreement_rate` values for each.

### Query 4 — Source-side dedup verification

```sql
SELECT count(*) AS n, command, target FROM ops_control_requests WHERE status IN ('pending','claimed') GROUP BY command, target HAVING count(*) > 1;
```

**Empty result = dedup working.** Any rows = dedup broken (same command accumulating without closure).

## Section 3 — Red Flags (Action Before A1 If Any)

Address these before shipping A1 if they appear:

**Query 4 returned any rows**
→ Source-side dedup is broken. The `ops_control_requests` table is accumulating duplicate pending
commands. Investigate:
- `src/lib/intelligence/oversight-agent.ts` `escalate()` — is it checking for an existing
  `pending` row before inserting?
- `src/lib/control-plane-db.ts` `createOpsControlRequest()` (or equivalent) — does it upsert
  or insert-blind?
Before A1 ships, the duplicate count must be 0. A miner trained on duplicate control_command
signals will propose a spurious rule.

**Query 2 returned 0**
→ `closeFinishedTasks` cron has not fired in 7 days (or never fired). The hub cannot auto-close
tasks without it; accumulation pressure builds. Investigate:
```bash
pm2 logs aria-bot --lines 500 | grep -i 'closeFinishedTasks\|CloseFinishedTasks'
```
Also check the `cron_runs` table:
```sql
SELECT job_name, status, started_at, completed_at FROM cron_runs
WHERE job_name ILIKE '%close%'
ORDER BY started_at DESC LIMIT 20;
```
If the cron is missing entirely, confirm `closeFinishedTasks` was registered in
`src/lib/intelligence/ops-manager.ts` and that `pm2 restart aria-bot` was run after the phase
2.5 hygiene migration was applied.

**Query 3 returned > 5**
→ The hub is accumulating open tasks faster than the cron closes them, or the spoke writers are
producing rows that have no `closes_when` predicate (so the cron can't evaluate them). Check:
```sql
SELECT type, source_table, closes_when IS NULL AS no_predicate, count(*)
FROM agent_task
WHERE status IN ('PENDING','NEEDS_APPROVAL')
GROUP BY 1, 2, 3
ORDER BY 4 DESC;
```
Rows with `no_predicate = true` are stuck indefinitely unless Will manually resolves them.
If the bulk are `control_command/restart_bot`, the hygiene backfill migration
`20260501_hygiene_backfill.sql` may not have been applied yet.
